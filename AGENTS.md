# AI 行为准则

## 一、任务执行原则

### 1. 理解优先
- 在编写代码前，先理解需求和现有代码结构

### 2. 增量修改
- 优先进行小而安全的改动，避免大规模重构除非必要

### 3. 测试验证
- 完成功能后运行相关测试确保正确性

### 4. 保持一致
- 遵循项目现有的代码风格和架构模式

## 二、决策优先级

1. 遵循现有代码风格和架构
2. 优先使用项目已有的工具和库
3. 考虑向后兼容性和 API 稳定性
4. 性能与可维护性的平衡

## 三、交流与反馈

- 明确说明代码改动的原因和影响
- 提供多个方案时，说明各方案优缺点
- 遇到不确定的问题时，主动询问确认
- 复杂变更前先获得用户确认

## 四、禁止事项

### 4.1 禁止的行为

| 禁止行为 | 说明 |
|----------|------|
| ❌ 不要修改用户明确标记为"不要修改"的文件 | |
| ❌ 不要提交包含敏感信息（API Key、Token、密码等）的代码 | |
| ❌ 不要进行未经确认的大规模重构（超过 10 个文件） | |
| ❌ 不要忽略编译错误或测试失败 | |
| ❌ 不要硬编码配置值，应使用配置文件或环境变量 | |
| ❌ 不要直接操作数据库，应通过 ORM 或 Repository | |
| ❌ 不要在主分支直接提交，应通过 Pull Request | |
| ❌ 不要在规范规则未完成前生成代码 | 必须等待用户确认规则文档已完成 |
| ❌ 绝对不要在未先更新需求文档的情况下修改或新增需求 | 必须严格按照以下流程：<br>1. 先更新 changelog.md 记录此次变更<br>2. 同步更新相关的需求文档（如 spec.md）<br>3. 然后才可以继续生成代码实现功能 |
| ❌ 不要在提交后忘记推送到远程仓库 | 每次 git commit 后必须立即执行 git push，确保代码及时同步到 GitHub |
| ❌ 绝对不要在未记录需求变更历史的情况下进行任何修改 | 所有修改必须先记录到 changelog.md，格式必须严格遵循：<br>- 时间格式：[YYYY-MM-DD HH:mm:ss]（必须包含时分秒）<br>- 必须填写：变更人、变更类型、变更原因、变更内容、影响范围、相关文档 |

### 4.2 安全红线

| 禁止行为 | 说明 |
|----------|------|
| ❌ 绝对禁止在代码中硬编码密码、密钥 | |
| ❌ 绝对禁止使用不安全的加密算法 | |
| ❌ 绝对禁止将用户隐私数据记录到日志 | |
| ❌ 绝对禁止跳过安全审计的代码提交 | |

### 4.3 性能红线

| 禁止行为 | 说明 |
|----------|------|
| ❌ 不要在主线程执行 I/O 操作（Android） | |
| ❌ 不要创建不必要的对象导致内存泄漏 | |
| ❌ 不要使用 SELECT * 查询 | |
| ❌ 不要忽略数据库索引设计 | |

## 五、变更流程

### 5.1 需求变更流程

```
1. 更新 changelog.md 记录变更
   └─> 时间格式：[YYYY-MM-DD HH:mm:ss]
   └─> 内容：变更人、变更类型、变更原因、变更内容、影响范围、相关文档

2. 同步更新相关需求文档
   └─> spec.md（需求规格）
   └─> 其他相关文档

3. 生成代码实现功能
   └─> 遵循现有代码风格
   └─> 编写测试用例

4. 运行测试验证
   └─> 确保编译通过
   └─> 确保测试通过

5. 更新Docker镜像并重新部署（必须执行）
   └─> **强制执行规则**：每次修改需求或代码后，必须自动更新Docker部署
   └─> 方式一：使用部署脚本（推荐，自动执行）
       - Windows: deploy.bat
       - Linux/Mac: ./deploy.sh
   └─> 方式二：使用docker-compose（手动执行）
       - docker-compose down
       - docker-compose up -d --build
   └─> 方式三：手动操作（适合调试）
       - docker stop <container_name>
       - docker rm <container_name>
       - docker build -t <image_name> .
       - docker run -d ... <image_name>

6. 提交代码
   └─> git commit
   └─> git push
```

### 5.2 changelog.md 记录格式

```
## [YYYY-MM-DD]

### Changed
- [YYYY-MM-DD HH:mm:ss] 变更人：变更内容（变更原因）
  - 影响范围：模块/文件
  - 相关文档：doc1.md, doc2.md
```

## 六、代码审查准则

1. **代码质量**
   - 代码符合编码规范
   - 有足够的测试覆盖率
   - 没有潜在的性能问题

2. **安全审查**
   - 没有硬编码的敏感信息
   - 使用安全的加密算法
   - 输入参数已校验

3. **文档更新**
   - 相关文档已同步更新
   - changelog.md 已记录变更

## 七、执行规范

本行为准则适用于所有参与本项目的AI代理，必须严格执行。任何违反准则的操作都可能导致代码质量问题、安全漏洞或项目管理混乱。
